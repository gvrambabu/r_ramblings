# MFE T1: ETF Order Ticket

Version: 1.0 | Owner: Trading Platform Squad | Persona: Trader

## 1. Purpose

The Order Ticket is the primary order-entry surface for fixed income ETF trades. It supports single-leg market, limit, and algo orders across DMA and RFQ execution protocols. It must be launchable in under 2 seconds and pre-populate from any FDC3 instrument broadcast. It is intentionally narrow in scope — it captures and submits; it does not show blotter history or P&L.

## 2. Personas & Entry Points

|Persona|How they access it            |Primary job-to-be-done              |
|-------|------------------------------|------------------------------------|
|Trader |Hotkey, FDC3 broadcast from T2|Enter and submit a new order quickly|
|Sales  |Launch from S1 Axes MFE       |Create a client-facilitated order   |

## 3. Layout & Component Breakdown

**Header Bar** (Shadcn `Card` header, `bg-zinc-900`)

- ETF ticker + full name (populated from FDC3 or X1 search)
- Side toggle: `BUY` (green accent) / `SELL` (red accent) — Shadcn `ToggleGroup`
- Close / minimise controls

**Core Entry Form** (2-column grid, Tailwind `grid grid-cols-2 gap-4`)

- Quantity field: shares or notional toggle (`Input` + `Select`)
- Order type: Market / Limit / VWAP / TWAP (`Select`)
- Limit price field: enabled only when Limit selected (`Input`, disabled state via Tailwind `disabled:opacity-50`)
- TIF (Time In Force): Day / IOC / FOK / GTD (`Select`)
- Execution venue: DMA / RFQ / Cross (`RadioGroup`)
- Broker/counterparty: (`Combobox`, pre-filtered by venue)

**Pre-trade Summary Strip** (full-width bar, `bg-zinc-800`, monospace font)

- Estimated notional · Spread to NAV · Last price · 30d ADV%
- Real-time — updates as quantity changes via AMPS subscription

**Action Bar**

- `Submit Order` primary button (Shadcn `Button variant="default"`)
- `Save as IOI` secondary button
- `Clear` ghost button

**Order Status Toast** (Shadcn `Toast`, appears on submit)

- Acknowledged / Rejected / Partially Filled states

## 4. Key Interactions

1. User opens ticket — FDC3 `fdc3.instrument` context auto-populates ticker, name, last price
1. User toggles BUY/SELL — side colour coding updates immediately across all fields
1. User enters quantity — pre-trade summary strip recalculates notional and ADV% in real time via AMPS
1. User selects “Limit” order type — limit price field becomes enabled and focused
1. User selects “RFQ” venue — counterparty field switches from broker list to RFQ dealer panel
1. User clicks Submit — optimistic UI shows “Submitting…” spinner; on ACK shows green toast; on REJECT shows red toast with reason code
1. On fill, FDC3 `fdc3.trade` context is broadcast to T2 Blotter and M1 Trade Capture

## 5. FDC3 Context

|Direction|Channel      |Context Type     |Payload summary                                |
|---------|-------------|-----------------|-----------------------------------------------|
|Inbound  |Green system |`fdc3.instrument`|`{ id: { ticker, ISIN, FIGI }, name }`         |
|Outbound |Green system |`fdc3.trade`     |`{ orderId, ticker, side, qty, price, status }`|
|Outbound |Purple system|`fdc3.order`     |Broadcast to M1 on confirmed submission        |

## 6. AMPS Pub/Sub Topics

|Direction|Topic                          |Trigger / Frequency                 |
|---------|-------------------------------|------------------------------------|
|Subscribe|`etf.quote.{ticker}`           |Real-time on ticker population      |
|Subscribe|`etf.pretrade.metrics.{ticker}`|On quantity change (debounced 300ms)|
|Publish  |`orders.new`                   |On Submit click                     |

## 7. Data Dependencies

- **Reference Data Service** — ETF metadata, ISIN/FIGI lookup
- **Quote Service (AMPS)** — real-time bid/ask, last, spread to NAV
- **Pre-trade Analytics Service** — notional calc, ADV%, estimated market impact
- **OMS Gateway** — order submission endpoint
- **Broker/Dealer Panel Service** — RFQ counterparty list

## 8. Constraints & Rules

- Notional hard limit: orders > $50M require a secondary confirmation modal
- Short sell: flagged automatically if position is flat/short (requires locate)
- Limit price must be within ±5% of last price (warn, not block)
- Venue selection must match counterparty eligibility matrix
- All fields must be locked during “Submitting” state to prevent duplicate submission

## 9. Empty / Error States

- **No instrument:** Full-bleed X1 Security Search component renders inline
- **Quote feed down:** Pre-trade strip shows `—` with amber “Delayed” badge; submission still permitted
- **OMS timeout:** Red toast “Order gateway unavailable — try again”; order not created
- **Loading:** Skeleton loaders on pre-trade strip fields only; form is interactive immediately

## 10. Open Questions

- [ ] Should VWAP/TWAP algo parameters (start time, end time, participation rate) live in this ticket or a separate Algo Parameters drawer?
- [ ] What is the SLA for OMS ACK before we show a timeout error? (Suggest 3s)
- [ ] Should Sales users see a “Client Account” field not shown to Traders?

# MFE T2: Live Blotter

Version: 1.0 | Owner: Trading Platform Squad | Persona: Trader · Middle Office (read)

## 1. Purpose

The Live Blotter is the real-time audit surface for all orders and executions in the current trading session. It gives traders a continuously updated view of order status, fills, and working orders so they can act immediately on partial fills, cancellations, or exceptions. It is the hub MFE — receiving FDC3 context from most other MFEs and broadcasting selection context back.

## 2. Personas & Entry Points

|Persona      |How they access it         |Primary job-to-be-done                  |
|-------------|---------------------------|----------------------------------------|
|Trader       |Pinned in primary workspace|Monitor order status, act on exceptions |
|Middle Office|Read-only snapshot view    |Verify trades ahead of T+0 capture in M1|

## 3. Layout & Component Breakdown

**Toolbar** (`flex justify-between`, `bg-zinc-900`)

- Filter chips: All · Working · Filled · Cancelled · Rejected (Shadcn `Badge` as toggles)
- Date range picker (today default) — Shadcn `DatePickerWithRange`
- ETF ticker filter — X1 Security Search inline (compact mode)
- Download CSV button

**Order Grid** (AgGrid, full viewport height minus toolbar)

- Columns: Time · Ticker · Side · Qty · Filled Qty · Avg Price · Order Type · Venue · Status · PnL (est.) · Trader
- Status column: colour-coded `Badge` — Working (blue) / Filled (green) / Partial (amber) / Cancelled (zinc) / Rejected (red)
- Row-level actions (right-click context menu): Cancel Order · Amend · Clone to New Ticket
- Pinned bottom summary row: total notional, total filled, session PnL

**Order Detail Drawer** (slides in from right, `w-96`)

- Full order lifecycle timeline (submitted → routed → acknowledged → fills)
- Individual fill ticks with time, qty, price, venue
- Linked RFQ quote chain if applicable

## 4. Key Interactions

1. New order arrives via AMPS `orders.updates` — row is inserted at top with flash animation (green for buy, red for sell)
1. Fill update arrives — Filled Qty and Avg Price update in-cell with highlight flash; status badge transitions
1. User clicks a row — Order Detail Drawer slides open; FDC3 `fdc3.instrument` broadcast on the green channel to sync T3 Ladder
1. User right-clicks → “Cancel Order” — confirmation popover appears; on confirm, cancel request published to `orders.cancel`
1. User right-clicks → “Clone to New Ticket” — T1 Order Ticket launches pre-populated with cloned order parameters
1. Filter chip toggled — grid filters in-memory (no re-fetch)

## 5. FDC3 Context

|Direction|Channel      |Context Type     |Payload summary                                |
|---------|-------------|-----------------|-----------------------------------------------|
|Inbound  |Purple system|`fdc3.order`     |New order broadcast from T1; row inserted      |
|Outbound |Green system |`fdc3.instrument`|Broadcast on row selection to sync T3, S2      |
|Inbound  |Green system |`fdc3.trade`     |Fill confirmation from T1; updates existing row|

## 6. AMPS Pub/Sub Topics

|Direction|Topic           |Trigger / Frequency                    |
|---------|----------------|---------------------------------------|
|Subscribe|`orders.updates`|Continuous — all order lifecycle events|
|Subscribe|`orders.fills`  |Continuous — execution reports         |
|Publish  |`orders.cancel` |On user-confirmed cancel action        |

## 7. Data Dependencies

- **OMS / Order Store** — initial session order load on mount
- **Execution Reports (AMPS)** — real-time fill stream
- **Reference Data** — ticker → full name resolution for display

## 8. Constraints & Rules

- Cancel is only permitted on orders in Working or Partially Filled status
- Amend is only permitted on Limit orders in Working status
- Middle Office view is read-only — context menu actions are hidden
- All timestamps displayed in local timezone with UTC offset shown on hover
- Grid must handle up to 10,000 rows without pagination (virtual scrolling via AgGrid)

## 9. Empty / Error States

- **No orders today:** Empty state illustration with “No orders in this session yet”
- **Feed disconnect:** Amber banner “Live feed interrupted — showing last known state”; reconnect auto-retried
- **Cancel rejected by OMS:** Red inline toast on the affected row with reason

## 10. Open Questions

- [ ] Should PnL column be estimated (vs NAV) or vs VWAP? Needs agreement with trading desk
- [ ] Do Middle Office users need a separate filtered view (their books only) or full desk view?

# MFE S1: Client Axes & Inventory

Version: 1.0 | Owner: Sales Technology Squad | Persona: Sales

## 1. Purpose

Client Axes & Inventory is the primary Sales workspace for managing the firm’s ETF inventory and distributing axes (indications of interest) to buy-side clients. It allows Sales to see what inventory is available, which clients have expressed interest in specific ETFs, and to initiate outbound axes directly from the interface. It is the Sales equivalent of the Trader’s blotter — always open, always live.

## 2. Personas & Entry Points

|Persona|How they access it       |Primary job-to-be-done                                 |
|-------|-------------------------|-------------------------------------------------------|
|Sales  |Pinned in Sales workspace|Find inventory, match to client axes, initiate outreach|

## 3. Layout & Component Breakdown

**Split Layout** (Tailwind `grid grid-cols-[2fr_1fr]`, resizable)

**Left Panel — Inventory Grid** (AgGrid)

- Columns: ETF Ticker · Name · Category · Available Size · Side (Long/Short) · Indicative Price · Spread · Last Updated
- Colour strip on row left edge: green (long/offer) / red (short/bid)
- Toolbar: filter by ETF category (IG Corp, HY, Muni, EM, Govt, Broad Market), search by ticker

**Right Panel — Client Axes**

- Tabbed: `My Clients` · `All Desk`
- List of client axis items: Client Name · ETF · Direction · Target Size · Urgency (High/Med/Low badge)
- Match indicator: amber highlight when an inventory row and a client axis share the same ETF ticker
- “Send Axis” button per client axis row — opens S2 RFQ Response Manager pre-populated

**Bottom Drawer — Axis Composer** (collapsible)

- Compose and blast an outbound axis to one or multiple clients
- Fields: ETF · Side · Size · Price / Spread · Expiry · Clients (multi-select)
- “Send to Selected” action → publishes to `axes.outbound` AMPS topic

## 4. Key Interactions

1. Page load — inventory grid populates from `inventory.etf` AMPS subscription
1. User clicks inventory row — right panel filters client axes to matching ETF; FDC3 `fdc3.instrument` broadcast to sync S2 RFQ Manager
1. Match highlight activates — amber badge on matched inventory row showing “N client axes”
1. User clicks “Send Axis” on client axis row — Axis Composer opens pre-filled with ETF and client
1. User composes and sends axis — published to `axes.outbound`; confirmation toast shown; axis marked “Sent” in client list
1. New inbound client inquiry arrives via `axes.inbound` — new row inserted in Client Axes panel with pulse animation

## 5. FDC3 Context

|Direction|Channel     |Context Type     |Payload summary                     |
|---------|------------|-----------------|------------------------------------|
|Outbound |Green system|`fdc3.instrument`|Broadcast on inventory row selection|
|Inbound  |Green system|`fdc3.instrument`|Sync from T2 Blotter selection      |

## 6. AMPS Pub/Sub Topics

|Direction|Topic          |Trigger / Frequency                    |
|---------|---------------|---------------------------------------|
|Subscribe|`inventory.etf`|Continuous — inventory position changes|
|Subscribe|`axes.inbound` |Continuous — new client inquiries      |
|Publish  |`axes.outbound`|On Axis Composer send                  |

## 7. Data Dependencies

- **Inventory Service** — real-time ETF inventory positions
- **CRM / Client Service** — client list, axis history, coverage assignments
- **Reference Data** — ETF metadata, category classification

## 8. Constraints & Rules

- Sales can only see inventory for their covered ETF categories (entitlements-controlled)
- Axis blasts to clients must not include size > risk-approved inventory
- Client list in Axis Composer is filtered to the Sales rep’s coverage list only
- Axes expire automatically after the specified expiry time; no manual deletion required

## 9. Empty / Error States

- **No inventory:** “Inventory feed unavailable” with retry button
- **No client axes:** “No active client axes for your coverage — check CRM”
- **Send failure:** Red toast with reason; axis remains in Composer for retry

## 10. Open Questions

- [ ] Should “All Desk” tab in Client Axes require explicit permission or be open to all Sales?
- [ ] Is the Axis Composer a drawer within S1 or should it launch as a separate small MFE window in OpenFin?

# MFE M1: Trade Capture & Confirmation

Version: 1.0 | Owner: Middle Office Squad | Persona: Middle Office

## 1. Purpose

Trade Capture & Confirmation is the Middle Office MFE for reviewing, enriching, and confirming trades flowing from the OMS. It is the first post-trade touchpoint — Middle Office reviews each trade, enriches it with allocation, legal entity, and settlement instructions, and sends it downstream to the confirmation and settlement systems. It must support both straight-through processing (STP) and manual exception handling.

## 2. Personas & Entry Points

|Persona      |How they access it            |Primary job-to-be-done                               |
|-------------|------------------------------|-----------------------------------------------------|
|Middle Office|Pinned in MO workspace        |Confirm trades, enrich exceptions, push to settlement|
|Trader       |Read-only link from T2 Blotter|Verify a specific trade has been captured correctly  |

## 3. Layout & Component Breakdown

**Status Summary Bar** (top, `bg-zinc-900`, full width)

- Count badges: Pending Capture · Pending Confirmation · STP’d · Exceptions · Confirmed
- Counts update in real time via AMPS

**Trade Queue Grid** (AgGrid, primary surface)

- Columns: Trade ID · Time · ETF Ticker · Side · Qty · Price · Notional · Counterparty · Status · STP Flag · Allocations
- STP Flag: green checkmark (clean STP path) / amber warning (requires enrichment) / red X (exception)
- Row colour coding: white (STP) / amber (enrichment needed) / red (exception)
- Inline edit: Allocation column editable in-grid for simple splits

**Trade Detail Panel** (right side drawer, `w-[480px]`)

- Trade header: all core economic terms (read-only)
- Enrichment form: Legal Entity · Settlement Account · Custodian · Allocation splits
- Timeline: trade lifecycle events with timestamps
- Action buttons: `Confirm & STP` · `Hold` · `Reject to Trader`

## 4. Key Interactions

1. New trade arrives via AMPS `trades.new` — row inserted with amber/green/red status immediately
1. STP trades auto-confirm after configurable delay (default 30s) unless user holds — countdown timer visible in row
1. User clicks exception row — Trade Detail Panel opens; enrichment fields highlighted in amber
1. User completes enrichment — field validation runs inline; `Confirm & STP` button activates
1. User clicks “Reject to Trader” — rejection reason modal appears; on submit, `trades.reject` published; T2 Blotter row updated
1. User confirms trade — status transitions to “Confirmed”; `trades.confirmed` published downstream

## 5. FDC3 Context

|Direction|Channel      |Context Type|Payload summary                               |
|---------|-------------|------------|----------------------------------------------|
|Inbound  |Purple system|`fdc3.order`|New order from T1; creates pending capture row|
|Outbound |Green system |`fdc3.trade`|Confirmed trade broadcast for reconciliation  |

## 6. AMPS Pub/Sub Topics

|Direction|Topic             |Trigger / Frequency                      |
|---------|------------------|-----------------------------------------|
|Subscribe|`trades.new`      |Continuous — new trades from OMS         |
|Publish  |`trades.confirmed`|On user confirmation                     |
|Publish  |`trades.reject`   |On reject to trader action               |
|Subscribe|`trades.updates`  |Lifecycle updates from downstream systems|

## 7. Data Dependencies

- **OMS / Trade Store** — source of new trades
- **Legal Entity Service** — entity list for enrichment
- **Settlement Instruction Service** — custodian / account lookup
- **Allocation Service** — fund allocation rules, validation
- **Downstream Confirmation System** — receives confirmed trades

## 8. Constraints & Rules

- Trades > $10M notional require dual-control confirmation (maker + checker)
- STP auto-confirm window is configurable per desk; default 30 seconds
- Rejected trades must include a reason code from a controlled vocabulary
- Allocation splits must sum to exactly 100%
- Confirmed trades cannot be amended — only a new correcting trade can be submitted

## 9. Empty / Error States

- **No pending trades:** “All caught up — no trades awaiting capture” with session summary
- **Enrichment validation failure:** Inline field-level errors, `Confirm & STP` button disabled
- **Downstream system unavailable:** Red banner; trades queue locally, confirmation held

## 10. Open Questions

- [ ] Is the 30s STP auto-confirm window configurable per-user or per-desk only?
- [ ] Does dual-control apply to all trades > $10M or only specific asset categories?
- [ ] Should rejected trades auto-notify the originating Trader via the X2 Notifications MFE?

# MFE X1: ETF Security Search (Shared)

Version: 1.0 | Owner: Platform Squad | Persona: All

## 1. Purpose

X1 is a shared utility MFE providing a fast, typeahead ETF security search used by all other MFEs. It runs as both a standalone launchable window and as an embeddable compact component (imported as a Module Federation remote). It broadcasts the selected instrument via FDC3 to all subscribing MFEs in the workspace.

## 2. Personas & Entry Points

|Persona|How they access it              |Primary job-to-be-done               |
|-------|--------------------------------|-------------------------------------|
|All    |Global hotkey / embed in any MFE|Find an ETF and set workspace context|

## 3. Layout & Component Breakdown

**Search Input** (Shadcn `Command` component)

- Typeahead on ticker, ISIN, CUSIP, full name
- Results grouped by category: IG Corp / HY / Govt / Muni / EM / Broad Market
- Result row: Ticker (bold) · Full Name · Category badge · Last Price · 1d Change%

**Recent & Watchlist** (below search, collapsible)

- Last 5 searched instruments
- User’s pinned watchlist (persisted in OpenFin workspace state)

## 4. Key Interactions

1. User types 2+ characters — debounced search (200ms) against reference data service
1. User selects result — FDC3 `fdc3.instrument` broadcast on green channel; all subscribing MFEs update
1. Standalone mode: window closes after selection (configurable)
1. Embedded mode: field populates in parent MFE; no window close

## 5. FDC3 Context

|Direction|Channel     |Context Type     |Payload                                            |
|---------|------------|-----------------|---------------------------------------------------|
|Outbound |Green system|`fdc3.instrument`|`{ id: { ticker, ISIN, CUSIP, FIGI }, name, type }`|

## 6. AMPS Pub/Sub Topics

None — X1 is read-only against Reference Data Service (REST).

## 7. Data Dependencies

- **Reference Data Service** — ETF universe, identifiers, metadata

## 8. Constraints & Rules

- Must return first results within 150ms of 2nd character typed
- Displays only ETFs (no single-name bonds, equities) unless filter is explicitly removed
- FIGI must always be included in broadcast payload for downstream system compatibility

## 9. Empty / Error States

- **No results:** “No ETFs found matching ‘{query}’” with suggestion to check ticker/ISIN format
- **Reference data unavailable:** Amber banner; search disabled; last used instrument retained in field
